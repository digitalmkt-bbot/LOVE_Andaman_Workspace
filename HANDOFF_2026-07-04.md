## What was built / changed

### 1. Client runs on SQL, no localStorage blob (`5e933e0`)
The full state (~5.7 MB) exceeded the browser localStorage quota → `setItem` threw
`QuotaExceededError`, which aborted the startup script before `window.LA_UID` was defined →
"LA_UID is not defined" cascade. Fix: the state blob lives in memory (`_mem` shim over
`localStorage.getItem/setItem` for the `loveandaman_v2` key). A reload always re-loads from SQL.

### 2. Per-entity REST API (`0a937f1`, extended in `cb999cc`)
`server.js` exposes generic CRUD over all 44 top-level entities, driven by the os_repo plan:
```
GET    /api/v1                    resource index {name: 'array'|'map'}
GET    /api/v1/<entity>[/<id>]    list / one record (nested children assembled)
POST   /api/v1/<entity>           create
PUT    /api/v1/<entity>/<id>      replace
DELETE /api/v1/<entity>/<id>      delete (children cascade)
POST   /api/v1/_batch             {baseVersion, ops:[...]} — many ops, ONE transaction
```
All writes go through `restApplyOp` (put/del/putall/meta) inside one transaction, serialized with
the legacy save via the same advisory lock, bump the `app_state` version once, and broadcast on SSE.
Requires `DATA_BACKEND=relational` (returns 503 otherwise — which is how it stays dormant on prod).

### 3. Saves are now per-entity, not whole-blob (`cb999cc`)
The client already computed a record-level diff per save. `laDiffToOps` translates that diff into
REST ops and sends them as one `POST /api/v1/_batch`. Editing one booking now writes ~15 rows
instead of re-inserting all 107 tables (~14k rows). **Fallback:** if a diff key isn't in the REST
index (mapping drift), the client logs a console warning and falls back to the legacy `/api/save`
whole-diff path — nothing breaks. `/api/load` stays a single bulk read (startup cache).

### 4. Mapper made fully lossless (`b5e5c0f`) — the big one
The `field_mapping.json` was auto-generated from a data **sample** = a whitelist. Every field/map-key
the app added since was silently dropped in relational mode (measured: 5,717 keys + 600 values on the
prod blob — route calendar overrides, docCheck, attachments, payment slips, seat-lock sub-groups,
vehicle day maps, market stats, contractHistory snapshots, fleet_daily trips).
- **open-ended maps** (date/id-keyed) → key-value **rows**, not per-key columns:
  `routes__overrides`, `sb_vehicles__dayroute/daystatus`, `sb_market_stats`, `nat_learn`,
  `insurance_overrides`, `vanjob_*`, `agent_artifacts`
- **new fields** → `json_text` columns (docCheck, attachments, paymentSlips, lockDraws, seat-lock
  fields, snapshot arrays…)
- `fleet_daily__trips` rebuilt as `(boat, key, value-JSON)` rows
- `os_repo` gained `map_value_json` kind + skips SQL NULLs (no fabricated `{a:null,…}` objects)

### 5. Smaller fixes
- Image previews (`att_...` 404) — attachments were copied into dev `public.attachments`; the app
  reads `allotment.attachments`. Recopied 159 files into the right table.
- Pickup Map default month → today (`523a9a8`) — was defaulting to the max trip date.
- Prod's 16 login users copied into `allotment.users`.

## Operational procedures (scripts in `os-backend/scripts/`)

DB URLs: `railway variables --project <id> --service Postgres --environment <env> --kv | grep DATABASE_PUBLIC_URL`
(prod project `df2af27d-...`, dev project `ff301670-...`).

**Mirror prod → dev (TWO parts — don't forget attachments):**
1. State: export the blob (`APP_STATE_ID=loveandaman_v2`, blobStore reads `public.app_state`) →
   `DATABASE_URL=<dev> DB_SCHEMA=operation_schemas node scripts/fast_import.js <blob.json>`
2. Attachments (files live OUTSIDE the blob): copy prod `public.attachments` → dev
   **`allotment.attachments`** (NOT public — else previews 404).

**Before importing / after any feature that adds fields — run the drift check:**
```
node os-backend/scripts/check_mapping_drift.js <fresh prod blob.json>
```
It fails loudly listing exactly what the mapper doesn't cover. If it flags something, add columns
following the pattern in `update_mapping_2026_07.js`, then `migrate_dev_2026_07.js` for the DDL.
**This is the maintenance habit that keeps the whitelist honest** — a new feature that adds a field
must be reflected in the mapper or its data is dropped (silently, without the check).

**Deploy dev:** `railway up --detach` from the workspace root.
