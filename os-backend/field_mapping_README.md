# app_state ⇄ operation_schemas — Field Mapping

This is the **reversible** mapping between the original `app_state` JSON (camelCase,
nested) and the flattened relational columns. It is generated by the actual
normalization script, so it is 1:1 with the loaded tables (1,397 columns, 103 tables).

Files:
- **`field_mapping.csv`** — one row per DB column: `table, db_column, db_type, app_source_path, kind`.
- **`field_mapping.json`** — same data keyed by `table → column → {source, kind, db_type}`.

## How the normalizer transformed names (and how to reverse it)

1. **Case fold / rename** — every key was lowercased and non-alphanumerics became `_`.
   `createdAt → createdat`, `rateTypeRef → ratetyperef`, `payType → paytype`.
   Reverse: use `app_source_path` in the mapping (it holds the original camelCase).

2. **Nested object un-flatten** — a nested object was inlined with an `_` join.
   `pax.ad_fr → pax_ad_fr`, `companyInfo.legalName → companyinfo_legalname`.
   The `app_source_path` shows the dotted original (`trips[].pax.ad_fr`), so you can
   rebuild the object by splitting on `.`.

3. **Arrays → child tables** — a nested array became its own table named
   `parent__field`, one row per element, linked by `<parent>_id` and ordered by `idx`.
   e.g. `bookings[].passengers[]` → `sb_bookings__passengers` (`sb_bookings_id`, `idx`).
   Reverse: `GROUP BY <parent>_id ORDER BY idx` and collect back into an array.

4. **Keyed maps → child tables** — an object used as a map (dynamic keys) became a
   child table with a `key` column holding the original key.
   e.g. `fleet_daily[date]` → rows with `key = <date>`.

5. **JSON-text columns** (`kind = json_text`) — a value that was itself an
   array-of-arrays / irregular structure was stored as a serialized JSON string.
   Reverse: `JSON.parse()` the column value. (This is the "different rule" case.)

## `kind` legend
- `scalar` — a normal leaf value (rule 1 / 2).
- `pk` — table primary key (`id`).
- `fk` — link to parent (`<parent>_id`), synthetic.
- `synthetic` / `synthetic-pk` — `idx`, `row_pk`: generated by normalization, **not** in the
  original JSON (drop these when writing back to app_state).
- `map_key` / `map_value` — original key / value of a keyed map (rule 4).
- `array_scalar` — element of an array of scalars.
- `json_text` — serialized JSON (rule 5).

## Rebuilding an app_state object from the tables (example: one booking)
```
booking = row of sb_bookings                      -- scalar + un-flatten (rules 1,2)
booking.passengers = sb_bookings__passengers      -- WHERE sb_bookings_id = booking.id ORDER BY idx
booking.trips      = sb_bookings__trips           -- same; re-nest pax_* back into pax{}
booking.addOns     = sb_bookings__addons
booking.history    = sb_bookings__history
```
Drop synthetic columns (`row_pk`, `idx`, `<parent>_id`) and re-apply the original key
names from `app_source_path` to get back the exact camelCase shape.
